# Sync ledger template

Shared reference for the resumable, auditable ledger written by every sync skill
after every run (success, partial failure, or zero-gap no-op):
`splunk-detector-publish` writes `.observe/detector-sync.md`;
`splunk-dashboard-publish` writes `.observe/dashboard-sync.md`. The shape is the same; only the object nouns and
the per-item columns differ.

A sync ledger makes a re-run idempotent: it re-reads live state, re-evaluates
each verdict fresh, and never re-creates something already present.

## Required structure

```markdown
# <Object> Sync Ledger: <service-name>

**Date:** <YYYY-MM-DD>
**Local spec:** `.observe/terraform/<file>.tf`
**Service filter resolved to:** `<service_name_value>`

## Summary

| Status | Count |
|--------|-------|
| COVERED | N |
| GAP → Created | N |
| GAP → Failed | N |
| UNCERTAIN | N |

## <Object> Status

| Local Spec | <Key> | Status | ID | Link | Reason |
|------------|-------|--------|----|----- |--------|
| ... | ... | COVERED | <id> | <app link> | <why — see Reason column> |

---
*Generated by <skill-name> on <YYYY-MM-DD>*
```

## The Reason column is required and must be non-empty

Every row — every COVERED, GAP, and UNCERTAIN verdict — must carry a **concrete,
non-empty Reason** stating *why* it was classified that way: which criteria fired
and what was compared. A generic note ("matched live object") is not acceptable;
the reason must name the live object and the exact match basis (or, for a GAP,
what was searched and found absent; for UNCERTAIN, the specific divergence). This
is the persisted form of the per-criterion rationale from
`coverage-decision-tree.md`.

Examples of acceptable Reason text:

- **COVERED** — `matched live dashboard "Checkout RED" (D-123): name match + 4/4 panel metrics present`
- **COVERED** — `metric http.server.request.duration + filter service.name=checkout + Standard origin all matched live detector "Checkout Latency" (id ABC)`
- **GAP** — `no live dashboard named "Checkout RED" in group "checkout"; will create`
- **GAP** — `no live detector with metric=db.pool.connections.active + filter service.name=checkout found`
- **UNCERTAIN** — `live dashboard "Checkout RED" exists but panels differ: local has p99_latency + error_rate; live has p50_latency only — needs human review`
- **UNCERTAIN** — `metric matches live detector but service filter absent in its programText — cannot confirm scope`

## Deep links

Use the realm's app URL for clickable links (see `splunk-api.md`):

- Detector: `https://app.${SPLUNK_REALM}.signalfx.com/#/detector/{id}`
- Dashboard: `https://app.${SPLUNK_REALM}.signalfx.com/#/dashboard/{id}`

## Incremental writes during chart creation

When creating objects in chart-first order, write (or overwrite) the ledger
file **after each successful chart POST** — not only once at the end. This
means a partial run that aborts mid-creation leaves the chart IDs persisted, so
a re-run can find and reuse them (or delete orphans) rather than creating
duplicates. The Step 7 final write is then a clean overwrite of the complete
state; the intermediate writes are a safety net.

## Idempotency

A re-run on a fully-synced service re-reads the live state and produces an
all-COVERED ledger whose reasons cite the matched live IDs — it does not
re-create anything because each verdict is re-evaluated fresh against live state,
not read from the prior ledger.
